# 定性评估场景清单（task 9.4）

自动化测试覆盖了结构正确性（模块拼装、断点形状、注入位置、可观测性断言）。
以下 5-6 个典型场景需接入真实 LLM（Anthropic / OpenAI）做人工对比定性评估，
验证「能干活 → 干得好」的行为层效果。每个场景记录：预期行为、实际行为、是否达标。

## 场景 1：缓存命中验证（Anthropic 路径）

- **操作**：启动 furflycode，连续多轮对话（≥3 轮），观察状态栏/日志的 token 用量。
- **预期**：第 1 轮 `cache_creation_input_tokens > 0, cache_read = 0`；第 2 轮起
  `cache_creation = 0, cache_read ≈ 第 1 轮 creation（±2%）`。
- **达标**：read 跨轮稳定，未出现 read 骤降（骤降即动静分离漏点）。

## 场景 2：FURFLY.md 规范生效

- **操作**：在项目根放一份 `FURFLY.md` 写入特定约定（如「所有新函数必须有 docstring」），
  启动后让模型写一个新函数。
- **预期**：模型生成的函数带 docstring，遵守 FURFLY.md 约定。
- **达标**：模型行为体现已读取并遵守 `<furfly_md>` 内规范。

## 场景 3：工具路由正确（hard_constraints 生效）

- **操作**：让模型查看某文件内容。
- **预期**：模型调用 `read_file` 而非 `bash` 里 `cat`；让模型改文件时先 `read_file` 再
  `edit_file`。
- **达标**：不出现「用 cat/sed/grep shell 命令做本应专用工具做的事」的回退；
  edit 前有 read。

## 场景 4：事件触发注入位置

- **操作**：构造 ≥8 回合的长对话，或发送含 `/plan` 的消息。
- **预期**：`<system_reminder>` 注入出现在 `messages` 末尾（日志可见），不在中间；
  同一事件不重复触发；模型未因注入进入过度防御姿态。
- **达标**：注入末尾追加、不击穿 system 缓存、模型行为不被频繁唤醒劣化。

## 场景 5：Plan Mode 缓存代价

- **操作**：在普通模式跑几轮后切 `/plan`，再切 `/do`。
- **预期**：每次切换后下一轮 `cache_read` 对 tools 部分出现一次 miss + rewrite
  （creation 短暂 >0）；Plan 模式下模型只输出计划不调写工具。
- **达标**：模式切换代价被观察到且符合预期；Plan 模式不升级子代理、行为受限。

## 场景 6：动静分离不被动态内容击穿

- **操作**：在不同工作目录启动会话（env_info 不同），对比同一项目的多轮缓存命中。
- **预期**：`system`/`tools` 缓存前缀不受工作目录变化影响（env_info 走 messages），
  跨会话静态前缀稳定。
- **达标**：环境变化不击穿 system 缓存，read 仍稳定命中。
